Архитектура и стек
Фреймворк: FastAPI (асинхронный, быстрый, встроенная Swagger-документация).
База данных: PostgreSQL (реляционная, отлично подходит для гео-данных, если в будущем прикрутить PostGIS).
Контейнеризация: Docker + Docker Compose.
ORM: SQLAlchemy 2.0 (или SQLModel).
Миграции: Alembic.
---
Блок 1: Инициализация и инфраструктура (Setup)
Цель: Поднять рабочий контейнер с базой и приложением, чтобы любой участник команды мог развернуть проект одной командой.
Задачи:
Написать Dockerfile для FastAPI.
Написать docker-compose.yml с двумя сервисами: web (FastAPI) и db (PostgreSQL).
Сделать базовый main.py с одним эндпоинтом GET /health (или /ping).
Настроить чтение переменных окружения (логины, пароли БД) через .env файл (используя pydantic-settings).
Критерий завершения (Тест): При запуске docker-compose up -d база поднимается, а по адресу http://localhost:8000/health возвращается {"status": "ok"}.
Блок 2: Проектирование БД и система миграций
Цель: Создать структуру данных и инструмент для ее обновления.
Задачи:
Инициализировать Alembic (alembic init).
Связать Alembic с базой данных в Docker.
Создать базовые модели:
User: id, email, role (citizen, moderator).
Issue (Проблема): id, user_id, title, description, latitude, longitude, image_url, status (pending_ai, approved, rejected, in_progress, resolved), priority (от 1 до 5).
Сгенерировать и применить первую миграцию.
Критерий завершения (Тест): В СУБД (например, через DBeaver) видны созданные таблицы users и issues.
Блок 3: Авторизация и Роли (Auth)
Цель: Разделить права доступа, чтобы обычные люди могли отправлять заявки , а модераторы — управлять ими.
Задачи:
Реализовать регистрацию и логин (выдача JWT токена).
Написать зависимости (Dependencies) в FastAPI для проверки токена: get_current_user и get_current_moderator.
Критерий завершения (Тест): Эндпоинт GET /users/me работает только с валидным токеном.
Блок 4: API для граждан (Загрузка проблем)
Цель: Позволить пользователям скачивать приложение и загружать свои проблемы с фото.
Задачи:
Эндпоинт POST /issues/: принимает текст, координаты.
Эндпоинт POST /issues/{id}/upload_image: загрузка фотографии (через UploadFile в FastAPI). Для начала файлы можно сохранять локально в папку (примонтированную как volume в Docker).
Эндпоинт GET /issues/my: список проблем текущего пользователя.
При создании проблемы ей автоматически присваивается статус pending_ai.
Критерий завершения (Тест): Пользователь с токеном может создать проблему, прикрепить фото и увидеть ее в своем списке.
Блок 5: Интеграция ИИ (Модуль приоритетов и фильтрации)
Цель: ИИ анализирует заявку, выставляет приоритет и удаляет (отклоняет) ложные вызовы.
Задачи:
Создать сервисную функцию analyze_issue_with_ai(issue_id).
На этапе разработки: Сделать мок (заглушку) нейросети, которая рандомно ставит приоритет или статус rejected.
На этапе интеграции: Подключить вызов реального API (например, OpenAI или локальной модели) для анализа текста проблемы.
Настроить вызов этой функции через BackgroundTasks в FastAPI при создании заявки (чтобы юзер не ждал ответа ИИ).
Критерий завершения (Тест): После создания заявки (Блок 4), ее статус через пару секунд меняется с pending_ai на approved с приоритетом от 1 до 5, либо на rejected (ложная заявка).
Блок 6: Панель Модератора (Akimat API)
Цель: Модератор (акимат) смотрит отфильтрованные ИИ проблемы и решает их.
Задачи:
Эндпоинт GET /admin/issues/: выдает список всех проблем.
Важно: Добавить фильтрацию (Query параметры) по статусам (например, скрыть rejected), сортировку по приоритету по убыванию.
Эндпоинт PATCH /admin/issues/{id}: изменение статуса проблемы (на in_progress или resolved).
Критерий завершения (Тест): Доступ к роутам /admin/... есть только у пользователей с ролью moderator. Модератор может поменять статус заявки на «решено».
Блок 7: Подготовка к модулю прогнозирования (Спецзадача)
Цель: Заложить фундамент для предикта пожаров и ЧС с помощью моделей. Эту часть можно выделить в отдельный домен.
Задачи:
Создать модель EmergencyPrediction: id, type (fire, infrastructure), probability (0.0 - 1.0), latitude, longitude, timestamp.
Эндпоинт POST /predictions/fire: принимает данные телеметрии (погода, датчики) и прогоняет через ML-модель для предсказания.
Эндпоинт GET /predictions/alerts: выдает список участков с высокой вероятностью возгорания для карты.
Критерий завершения (Тест): При отправке mock-данных о сильном ветре и сухой погоде, API возвращает высокий процент вероятности пожара.
